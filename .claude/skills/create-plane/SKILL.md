# Create Plan — 規劃功能的方法論

## 何時使用此 skill

當使用者出現以下情境時觸發：
- 出示系統架構圖並詢問接下來要建構什麼
- 引用規格文件（如 `sign_off_system.md`）並要求依此規劃
- 說出「進入規劃階段」、「重新檢視計畫」、「幫我確認下一步」
- 要求對照現況實作與目標架構
- 想在實作前取得分階段開發路線圖

**不要在以下情境觸發此 skill：** 使用者詢問如何實作某個具體元件 — 那是執行階段，不是規劃階段。此 skill 產出計畫，不產出程式碼。

## 規劃方法論 — 7 個階段

### 階段 1 — 盤點現有程式碼

動筆前先調查現況。使用 Explore subagent 並行搜尋以擴大涵蓋面：
- `lib/page/` — 已存在哪些頁面模組
- `lib/service/` — 已存在哪些服務
- `lib/model/` — 已存在哪些資料模型
- `lib/repositories/` — 已存在哪些 repository
- 首頁側邊欄 — 哪些已串接、哪些只是佔位
- 路由清單 — 哪些路由已註冊

產出：一份扁平的清單，列出既有模組、路徑、明顯的佔位項目。

### 階段 2 — 架構圖對照程式碼

對架構圖中**每一個方塊**，產出一列狀態表：

```
| 模組 | 狀態 | 對應路徑 |
|------|------|---------|
| 組織規劃 | ✅ | page/org_design/org_manager/ |
| 簽核級別 | ❌ | 無對應 page/service/model |
| 發起簽核 | ⚠️ | form_application_center/ 部分完成，缺 X |
```

狀態圖示：
- ✅ 完成 — 模組已存在且可運作
- ⚠️ 部分完成 — 模組存在但缺關鍵部分；需描述缺口
- ❌ 未開始 — 沒有程式碼，可能僅有選單佔位

**依架構圖列分組**（如：第一列：動態表單屬性、第二列：權限設置）。

### 階段 3 — 與規格文件交叉比對

針對每個 `❌` 或 `⚠️` 方塊，找出規格文件中對應的章節。引用或改寫規格內容，確認該方塊的設計意圖。

此階段常會發現**重要釐清** — 架構圖與規格書互相矛盾、或兩個方塊其實是同一功能換了名稱。將這些以醒目區塊標出：

```markdown
> **重要釐清：** 架構圖中「發起設定」與「表單發起權限」是同一功能。
> 對照 sign_off_system.md 的「建立簽核流程模板」章節 ...
```

### 階段 4 — 將缺口拆分為階段

將待開發模組分組為 Phase A、B、C，依據：
- 架構圖順序（由上而下、由左而右）
- 依賴關係（B 依賴 A 的 model）
- 邏輯內聚（相關模組放同一 Phase）

每個 Phase 標題應包含：
- 對應架構圖：哪些方塊
- 對應規格書：哪些章節
- 目標：一句話說明
- 整合：與既有模組的依賴關係

### 階段 5 — 細化每個階段

每個 Phase 章節內應包含：

1. **核心設計** — 高階設計決策
   - Enum 與其值
   - 概念流向（如「三種簽核流向：垂直向上 / 同層互送 / 跨業務線」）
   - 概念→識別碼→使用情境的對照表
   - 一個具體的範例走過完整流程

2. **資料模型** — 以 ASCII 樹狀圖表示，不要寫 Dart 程式碼
   ```
   ApprovalCanvasNode
   ├── nodeId          : String
   ├── offsetDx        : double
   └── nodeType        : NodeType
   ```

3. **需要建立的檔案** — 依層次分組的具體檔案清單：
   - Enums（`lib/enum/`）
   - Models（`lib/model/`）
   - Repositories（`lib/repositories/`）
   - Services（`lib/service/`）
   - Pages（`lib/page/{feature}/`）
   - Widgets（在 page 的 widgets/ 下）

4. **與現有模組銜接** — 依賴對照表：
   ```
   | 現有模組 | 銜接方式 |
   | form_launch_permission | 提供發起資格設定，被流程模板引用 |
   ```

5. **路由與選單** — 在側邊欄與路由表中的位置

### 階段 6 — 排序工作順序

產出樹狀的開發順序。每個步驟要小到能成為檢查點：

```
Phase A
  ├── A1: Enums + Models + Repository
  │     ├── ApproverSourceType / NodeType / MultiStrategy enums
  │     ├── ApprovalWorkflowTemplate / ApprovalNode models
  │     └── LocalStorage Repository
  ├── A2: Service（含 resolveApproverChain）
  ├── A3: 流程模板列表頁
  ├── A4: 流程編輯器
  │     ├── 4.1 三欄 Scaffold + Tab 切換
  │     ├── 4.2 發起設定 Tab
  │     └── 4.3-4.6 ...
  └── A5: 首頁選單整合 + 路由註冊
```

原則：**每個葉節點步驟可獨立驗證**（編譯、手動 smoke test、或單畫面 demo）。

### 階段 7 — 技術一致性 + 驗證 + 待確認事項

收尾三個小節：

1. **技術架構一致性** — 確認新工作沿用既有模式：
   - State Management：BLoC + Equatable
   - Storage：LocalStorage（未來轉 API）
   - Navigation：GoRouter + RouteName
   - Theme：ThemeExtension 模式
   - DI：GetIt service locator
   - UI Pattern：漸層背景 + GlowOrb + Shell + Panel

2. **驗證方式** — 使用者如何驗證完成度：
   - `flutter analyze --no-pub` 回報 0 errors
   - 手動走過完整使用者流程
   - 每個 Phase 對應的 smoke test

3. **待確認事項** — 開放設計決策的 checkbox 清單：
   ```
   - [ ] 「向上 N 層」相對基準：相對申請人 vs 相對前一節點？
   - [ ] 加簽功能：v1 是否實作？
   ```

   **每一項都要附上建議方案**（如「建議方案：相對申請人」），讓使用者能快速接受或推翻。

## 文件結構模板

```markdown
# {功能名} 規劃 — {副標}

## 規劃背景
### 已完成項目總覽
| 子系統 | 模組 | 狀態 |
### 流程位置 / 進度
[ASCII 流程圖顯示此功能在整體系統中的位置]

## 一、要解決什麼問題
[情境表 / 痛點表]

## 二、釐清模糊地帶
[架構圖與規格書的交叉比對；標出重複或重疊]

## 三、核心設計
[Enum、類型、概念流向、具體範例]

## 四、資料模型
[每個 model 的 ASCII 樹狀圖]

## 五、需新增的模組
[每個模組：定位 / 功能表 / 路由]

## 六、與現有模組的銜接
[依賴關係表]

## 七、檔案結構
[lib/ 下新增檔案的樹狀結構]

## 八、BLoC 事件設計
[依分類整理的事件清單]

## 九、建議開發順序
[樹狀的步驟與子步驟]

## 十、待確認事項
[含建議方案的 checkbox 清單]

## 十一、後續階段
[若此為 phase A，簡述 phase B/C]
```

## 套用此 skill 的步驟

1. **從使用者收集輸入：**
   - 架構圖（圖檔或文字描述）
   - 規格文件路徑（如 `docs/system_docs/` 下的 markdown）
   - 其他額外限制

2. **執行階段 1**（盤點程式碼）使用 Explore subagent 並行搜尋。

3. **執行階段 2**（對照表）— 先寫架構圖對照程式碼的狀態表；這份表是後續所有規劃的錨點。

4. **執行階段 3**（交叉比對）— 找出架構圖與規格書之間的重複或命名不一致，明確標出。

5. **執行階段 4-6**（拆分、細化、排序）— 寫到 `docs/planning/{feature}_planning.md`。

6. **執行階段 7**（一致性、驗證、待確認）— 以開放問題清單收尾。

7. **停下來，不要寫程式碼。** 等使用者檢視計畫並回答開放問題。

## 產出規則

- 寫入 `docs/planning/{feature}_planning.md`（相對於專案根目錄）
- 標題與表格內容使用繁體中文（符合專案慣例）
- 程式碼區塊內的檔案路徑要相對於專案根目錄
- 編譯檢查統一使用 `flutter analyze --no-pub`
- 引用規格時逐字標出 `docs/system_docs/...` 路徑
- 深度對齊 `docs/planning/approval_workflow_planning.md` — 不要過淺也不要灌水

## 反模式警示

- **不要寫實作程式碼** — 此 skill 只產出計畫
- **不要省略盤點階段** — 對既有程式的假設通常是錯的
- **不要為了好看而合併 Phase** — 獨立可驗證性比行數重要
- **不要省略「待確認事項」** — 即使是「完美」計畫也會有 3-5 個真實開放問題
- **不要只列問題不給建議** — 每個問題都要附「建議方案」
- **不要在沒實際比對的情況下宣稱一致性** — 實際讀一個既有模組來驗證 BLoC/Theme/DI 模式

## 使用者反饋處理

常見的修正請求與對應行動：

| 使用者說 | 對應行動 |
|---------|---------|
| 「X 與 Y 是同一功能」 | 加上「重要釐清」區塊、在計畫中合併、若不確定則放入「待確認事項」 |
| 「改用拖曳/視覺化操作」 | 只更新 Section 五（模組）與 Section 七（檔案結構）；資料模型保持穩定 |
| 「v1 不要做 X」 | 把 X 移到「待確認事項」→「後續階段」；不要刪除，只降級 |
| 「再給一個範例」 | 補在 Section 三（核心設計）— 具體走查很便宜且能釐清意圖 |
| 「這個 Phase 太大」 | 拆開。「建議開發順序」中的每個葉節點應 ≤1 天工作量 |

## 最後提醒

計畫文件就是契約。若使用者批准後請求實作，實作必須符合計畫，否則必須明確更新計畫。不要默默偏離。
