
# 電子簽核系統 發想
[toc]
## 主要目的
> 目標：設計一套 **可共用、可配置、可擴充 與 快速部屬** 的電子簽核前端系統  
> 適用場景：行政 / 採購 / 請假 / 費用 / 用印 / 內部申請流程

本系統本質上由 **以下核心子系統** 組成：
1. **組織架構與權限底層（Foundation）**
   - 定義公司組織、人員、角色與代理關係
   - 人員權限
2. **動態表單系統（Dynamic Form System）**
   - 動態「申請資料本身」的結構與規則
   - 配合API，拉取對應條件Form。
3. **流程引擎（Workflow / Approval Engine）**
   - 根據規則產生簽核路徑、推進流程、處理例外。
4. **歸檔**
   - 完成簽核
   - 紀錄事件，表單追蹤


> 共用性的關鍵在於：透過MetaData 透過映射方式
> - 組織 / 角色 / 權限可動態配置
> - 表單可動態配置
> - 流程可動態配置


## 組織架構與權限底層（Foundation）
「表單發起限制、簽核人解析、代理/轉派、稽核追溯」皆以本層資料為準。  
1. 身份模型
2. 公司組織樹
3. 代理人矩陣

### 身份模型
目的： 明確定義「系統中的人是誰、可以做什麼事」
- 使用者（User）
  - 唯一識別（UserId）
  - 狀態（在職 / 停用 / 離職）
  - 所屬角色（可多重）
  - 所屬組織關係（可多重）
- 角色（Role）
  - 系統角色：申請人 / 簽核人 / 管理者 / 稽核
  - 業務角色：採購人員 / 財務 / HR / 行政 / 專案負責人
- 權限範圍（Scope）
  - 公司 / 部門 
  - 表單類型 / 流程類型（可選）
### 公司組織樹
**目的：** 支援簽核人解析的「上下級關係」與「責任歸屬」。
- 行政隸屬（Department Hierarchy）
  - 公司 → 事業群 → 部門 → 小組
  - 支援
      - 直屬主管解析
      - 向上 N 層
### 代理人矩陣
**目的：** 避免簽核因請假、出差、離職造成流程中斷。
- 全域代理（Global Delegation）
  - 某人所有簽核任務在指定期間內轉交給代理人
  - 例：主管休假，所有單子轉給 A
- 特定代理（Scoped Delegation）
  - 僅針對特定範圍生效：
    - 表單類型（採購 / 請假）



## 基礎簽核流程定義
1. 表單建立
2. 設定簽核流程
3. 執行簽核
4. 各層級簽屬
5. 申請表單追蹤

### 表單建立
**目的：** 建立「可送簽」的表單入口，並確保符合制度規範。

- **表單模板（Template）**
    -  表單名稱 / 類型（行政、採購、請假…）
    - 欄位定義：金額、明細表、附件、人員/部門選擇
        - 元件模組： 文字、日期、數字、下拉。
        - 輸入元件： TextField 預設框架修改。
    - 驗證規則：必填、格式、範圍、跨欄位（例如結束日期 ≥ 開始日期）
    - 欄位可見性：依角色/條件顯示（例如勾選「需採購」才顯示金額）
    - 版本管理：模板版本 v1/v2（避免流程中途變更造成稽核問題）
- 表單種類建立
    - 請假申請單
    - 費用報銷單
    - 採購申請單
    - 用印申請單
    - 加班申請單
    - 人事異動申請單

```
表單建立（使用者自行建立）
    ↓
資料連結（欄位綁定資料來源，表單有內容）
    ↓
發起權限控管（在有內容的表單上設定誰可發起）
    ↓
簽核流程（知道誰發起、填什麼，才解析簽核路徑）
```



### 權限&設定簽核流程

**目的：** 事先定義公司制度中「這類申請要怎麼簽、由誰負責」。

- **建立簽核流程模板（Workflow Template）**
  - 發起資格 (權限相關)：
    - 角色限制（例：採購人員）
    - 部門限制
    - 身份條件（在職 / 職級 / 試用期）
    - 條件式組合（AND / OR）
  - 流程節點定義：
    - 審核節點（Approve / Reject / Return）
    - 會簽節點（多人：全部同意 / 任一同意）
    - 知會節點（不影響流程，僅通知留痕）
    - 條件分支（依金額、天數、類型、部門）
  - 簽核人來源：
    - 直屬主管（向上 N 層）
    - 指定角色（HR / Finance / 採購）
    - 部門主管
    - 表單欄位指定人 
  - 節點規則：
    - 簽核人來源：直屬主管N層、角色（HR/Finance）、部門主管、表單欄位指定人
    - 多人策略：AND / OR / 依序
    - SLA：逾時提醒、逾時升級
    - 退回策略：退回申請人 / 退回上一關 / 退回指定關卡
- **流程狀態：**
  - 設計中（Draft）
  - 啟用中（Active）
  - 停用（Disabled）



### 執行簽核
**目的：** 依既定流程推進簽核，不依賴人工指定。

- 申請單送出時：
  - 驗證發起資格
  - 解析並凍結簽核流程路徑
  - 產生對應簽核任務
- 確認簽核節點:
  - 顯示此表單需簽核的對象
- 簽核任務狀態：
  - 待處理 
  - 已同意 
  - 已拒絕 
  - 已退回 
  - 已逾時 
  - 已取消


### 各層級簽屬
**目的：** 確保每一關決策具備可追溯性與不可否認性。

- 每次簽核必留：
  - 動作類型（同意 / 拒絕 / 退回…）
  - 簽核意見（文字 / 結構化原因）
  - 附件（補件 / 佐證）
  - 操作者、角色、時間戳
- 簽核動作：
  - 同意 → 進下一節點或結案
  - 拒絕 → 流程終止
  - 退回 → 依規則回到指定節點
  - 補件要求 / 代理 / 轉派 (發錯人處理) / 加簽（需留痕）
- 簽名層級（依需求）
  - 系統簽核
  - 手寫簽名
  - 數位簽章
- 稽核原則：
  - 歷程僅能追加，不可修改或刪除
  - 管理者介入需記錄原因與權限

### 申請表單追蹤
**目的：** 提供透明的流程進度與完整歷程。
- 申請人視角：
  - 目前關卡與待處理人
  - 完整歷程
  - 補件待辦
  - 結案結果與封存資料
- 管理/主管視角：
  - 待簽與逾時總覽
